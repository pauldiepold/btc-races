# BTC Wettkampfanmeldung — Design Context

## Design Context

### Users
Vereinsmitglieder des Berlin Track Club (~100 Athleten) und Admins (~10 Coaches/Verantwortliche). Nutzungskontext: mobil und desktop, kurze Task-Flows (Anmelden, Status checken). Hohe Instagram-Affinität — die Zielgruppe hat einen ausgeprägten visuellen Anspruch. Admins brauchen Überblick und Effizienz bei der Verwaltung.

### Brand Personality
**Jung. Leistungsorientiert. Menschlich.**

BTC ist das Gegenteil eines altmodischen deutschen Leichtathletik-Vereins: dynamisch, community-getrieben, offen für alle Schnellen. Die Vision ist Laufen als Mannschaftssport — nicht Wettkampf gegen andere, sondern mit anderen. Visuell stark auf Instagram präsent.

Anti-Referenz: konservative, trockene Vereinswebsites. Generisches Nuxt-UI-Grau.

### Aesthetic Direction
**Clean Black & Yellow.** Echtes Tief-Schwarz als Basis (zinc-950, `#09090b`), BTC Yellow (#ffb700) als Akzentfarbe — kräftig und präsent, nicht dekorativ. Klare Kontraste, viel Weißraum, minimale visuelle Unruhe. Denk Nike-Produktseite, nicht Vereinshomepage.

Dark Mode ist der primäre Modus. Light Mode ist vollwertig umgesetzt, aber nicht der Fokus.

### Implementierte Tokens

| Token | Light | Dark |
|-------|-------|------|
| `--ui-primary` | yellow-600 | yellow-400 |
| `--ui-bg` | white | zinc-950 |
| `--ui-bg-muted` | zinc-50 | zinc-900 |
| `--ui-bg-elevated` | white | zinc-900 |
| `--ui-bg-accented` | zinc-100 | zinc-800 |
| `--ui-radius` | 0.25rem | ← |

### Typografie
- **Body / UI:** Raleway (400, 500, 600, 700) — geometrisch, klar, eigenständig
- **Display / Headings (h1–h4):** Space Grotesk (500, 600, 700) — modern, präzise, sporttauglich; aktiviert via `.font-display`-Klasse oder `h1`–`h4`-Tags

### Design Principles
1. **Kontrast vor Dekoration** — Gelb wirkt durch Kürze, nicht durch Fläche. Eingesetzt als Akzent, nicht als Füllung.
2. **Sportliche Klarheit** — Kein visuelles Rauschen. Jedes Element rechtfertigt sich durch Funktion.
3. **Community first** — Die Teilnehmerliste, Status-Badges und Anmeldeflüsse sind die Kerninteraktion. Diese Elemente bekommen die stärkste visuelle Energie.
4. **Mobile-first, aber kein Kompromiss** — Die meisten Nutzer sind mobil, aber Admin-Flows brauchen Desktop-Densität.
5. **Dezent anders** — Erkennbar als "nicht generisch" ohne zu schreien. Der BTC-Charakter soll spürbar sein, nicht explizit.

## Navigation & Layout-Architektur

### Navigation-Pattern

**Keine Haupt-Navigation.** Keine Nav-Leiste im Header, keine Bottom-Nav, keine Tab-Leiste.

- **Startseite (`/`)** ist die Event-Liste — der einzige reguläre Einstiegspunkt für Mitglieder
- **Meine Anmeldungen** sind über das UserMenu erreichbar (kein eigener Nav-Link)
- **Admin-Bereich** ist über einen Link im Footer erreichbar — nur sichtbar für `role === 'admin'`
- Navigation geschieht über Kontext: von der Event-Liste in eine Event-Detailseite, von dort zurück

Rationale: ~100 Nutzer, ein primärer Flow (Events anschauen → anmelden). Haupt-Nav wäre für die tatsächliche Komplexität der App oversized.

### App-Shell (`app.vue`)

```
UHeader
  #title   → BTC-Logo + "Berlin Track Club"
  #right   → UserMenu (Desktop)
  #body    → UserMenu expanded (Mobile Drawer)

NuxtLoadingIndicator

UMain
  NuxtPage

UFooter
  #left    → UColorModeButton (für nicht eingeloggte User / Fallback)
  #center  → © Berlin Track Club
  #right   → GitHub-Link + [Admin-Link, nur wenn role === 'admin']
```

### UserMenu-Struktur

```
[Yellow accent stripe]
[Avatar + Name + E-Mail]
─────────────────────
[Meine Anmeldungen →]   ← Link zu /meine-anmeldungen
─────────────────────
[Darstellung: Light | Dark]
─────────────────────
[Ausloggen]
```

Der Color-Mode-Toggle im Footer (`UColorModeButton`) bleibt als Fallback für nicht eingeloggte Nutzer (Login-Seite).

### Komponenten-Philosophie

**Keine generischen Seiten-Wrapper.** `BasePage` und `BaseLayer` (aus v1-Überresten) werden nicht weitergebaut. Jede Seite baut ihr Layout direkt mit `UContainer`, Nuxt UI Primitives und seitenspezifischen Strukturen. Drei ähnliche Zeilen Code sind besser als eine voreilige Abstraktion.

## Status-Badge-Farbtokens

Für Anmelde-Status-Badges. Farben passen ins Black/Yellow-System — nicht zu grell, klare Semantik.

| Status       | Light              | Dark               | Semantik                          |
|--------------|--------------------|--------------------|-----------------------------------|
| `registered` | `green-600`        | `green-400`        | Bestätigt / aktiv angemeldet      |
| `pending`    | `sky-600`          | `sky-400`          | Wartet auf Admin-Bestätigung      |
| `maybe`      | `amber-600`        | `amber-400`        | Interesse bekundet (Training etc) |
| `canceled`   | `red-600`          | `red-400`          | Abgemeldet / storniert            |
| `waitlisted` | `zinc-500`         | `zinc-400`         | Auf Warteliste                    |

**Wichtig:** `amber` (nicht `yellow`) für `maybe` — Yellow ist die Brandfarbe, darf nicht mit Status-Semantik kollidieren.

Umsetzung: `UBadge` mit `color`-Prop oder direkten Tailwind-Klassen. Kein eigenes Badge-Wrapper-Komponent nötig.
