# UX-Redesign: EventRegistrationList — Tab-basierte Anmeldungsliste

_Stand: 2026-04-05 | Kontext-Session: 9.6 (Event-Detail)_

---

## Ausgangssituation

Die Komponente `app/components/event/EventRegistrationList.vue` wurde im Rahmen von Session 9.6 als erste funktionierende Version implementiert. Sie zeigt alle Anmeldungen als flache Liste: aktive oben (ohne Unterscheidung zwischen Statusgruppen), abgemeldete kompakt als durchgestrichener Kleintext unten.

### Probleme der aktuellen Version

1. **Statusmischung**: "Angemeldet", "Dabei" und "Vielleicht" landen ungetrennt in einem Block. Das Verhältnis (z.B. 5 bestätigt / 3 vielleicht) ist nicht auf einen Blick erkennbar.
2. **Abgemeldete als Nachgedanke**: Durchgestrichener Kleintext am Ende des Blocks hat kein visuelles Gewicht. Die Zahl der Abgemeldeten ist nicht sofort sichtbar.
3. **Kein Scan-Modus**: Karten-Layout (`border bg-muted` pro Person) ist bei 15+ Einträgen eine Scroll-Wand.
4. **Status-Badge ist redundant** innerhalb einer Statusgruppe — wenn alle in der Liste "Angemeldet" sind, braucht man kein Badge mehr pro Zeile.

---

## Ziel der Überarbeitung

Eine **tab-basierte Ansicht**, bei der jede Statusgruppe einen eigenen Tab mit Zähler bekommt. Maximal 3 Tabs, abhängig vom Event-Typ. Kompakte Listenzeilen statt Karten. Für normale Mitglieder konzipiert — sie wollen wissen, wer kommt.

Admin-spezifische Funktionen (LADV-Operationsstatus, Aktions-Buttons) kommen in einer späteren Session (geplant bei 9.9) und werden in diese Struktur dann eingebettet.

---

## Status-Modell (aus 03-feature-spec.md)

Das Statusmodell kennt je nach Event-Typ genau 2–3 mögliche Werte, die niemals gemischt werden:

| Event-Typ     | Mögliche Status-Werte                   |
|---------------|------------------------------------------|
| `ladv`        | `registered` \| `canceled`              |
| `competition` | `registered` \| `maybe` \| `canceled`  |
| `training`    | `yes` \| `maybe` \| `no`                |
| `social`      | `yes` \| `maybe` \| `no`                |

**Konsequenz:** Niemals mehr als 3 Tabs. Die Tab-Labels sind event-typ-spezifisch.

---

## Tab-Mapping

| Event-Typ     | Tab 1 (positiv)         | Tab 2 (unsicher)   | Tab 3 (negativ)     |
|---------------|-------------------------|--------------------|---------------------|
| `ladv`        | Angemeldet              | —                  | Abgesagt            |
| `competition` | Angemeldet              | Vielleicht         | Abgesagt            |
| `training`    | Dabei                   | Vielleicht         | Nicht dabei         |
| `social`      | Dabei                   | Vielleicht         | Nicht dabei         |

- Tab 1 ist beim Laden immer aktiv (der positive Status ist das primäre Interesse der Nutzer)
- Tabs mit 0 Einträgen werden trotzdem gezeigt (damit man sieht, dass z.B. niemand abgesagt hat) — AUSNAHME: `ladv` hat keinen "Vielleicht"-Tab
- Der Zähler auf jedem Tab zeigt die Anzahl der Einträge in dieser Gruppe

## Neue Komponenten-API

### Neue Props

```ts
defineProps<{
  registrations: RegistrationDetail[]
  eventType: 'ladv' | 'competition' | 'training' | 'social'
}>()
```

Der `eventType` ist in `EventDetail` (Typ: `Event['type']`) vorhanden und wird von der aufrufenden Seite übergeben.

### Übergabe in `app/pages/events/[id].vue`

Aktuell (Zeile 81):
```html
<EventRegistrationList :registrations="event.registrations" />
```

Neu:
```html
<EventRegistrationList
  :registrations="event.registrations"
  :event-type="event.type"
/>
```

## Visuelles Design

### Tabs

Nuxt UI `UTabs`-Komponente. Variante `pill` oder `link` — im BTC-Design-System werden Tabs typischerweise als `pill` gerendert. Der aktive Tab soll klar hervorstechen.

Tab-Item-Struktur (als `items`-Array):
```ts
{
  label: 'Angemeldet',
  badge: '8',  // Anzahl als Badge-String
  slot: 'registered'
}
```

Alternativ: Tab-Label mit inline Zähler, z.B. `Angemeldet (8)` — einfacher, kein Badge-Prop nötig. **Bevorzugt**, da weniger Konfiguration.

### Listenzeilen (statt Karten)

Aktuell: jede Person bekommt eine eigene Box mit `border bg-muted rounded px-4 py-3`.

**Neu: kompakte Zeilen**, getrennt durch leichte Divider (`divide-y divide-default`), kein eigener Container pro Person.

Struktur pro Zeile:
```
[Avatar Initiale 32px]  [Name]  [Disziplin-Badges, wenn vorhanden]
                         [Notiz, wenn vorhanden — kursiv, gedämpft]
```

- Avatar-Initiale bleibt (32×32px, `bg-accented`, rund) — gibt der Liste Gesicht
- Name: `text-sm font-medium text-highlighted`
- Disziplinen-Badges: wie bisher, `color="neutral" variant="outline" size="xs"`
- Notiz: `text-xs text-muted italic`
- **Kein Status-Badge** mehr pro Zeile — der Tab definiert den Status bereits

### Leerzustand pro Tab

Wenn ein Tab keine Einträge hat:
```
[kleines Icon]
Noch niemand in dieser Gruppe.
```
Zentriert, `text-sm text-muted`. Kein aufdringlicher Leer-Zustand — der ist nur relevant wenn Tab 1 (positiver Status) leer ist.

### Gesamter Leerzustand (alle Tabs leer)

Wenn `registrations.length === 0`:
Bestehender Leerzustand bleibt — Icon + "Noch keine Anmeldungen." Das Tabs-UI wird in diesem Fall gar nicht gerendert (gleiches Verhalten wie bisher).

## Implementierungsdetails

### Computed: Tab-Konfiguration

```ts
const tabConfig = computed(() => {
  if (props.eventType === 'ladv') {
    return [
      { key: 'registered', label: 'Angemeldet' },
      { key: 'canceled', label: 'Abgesagt' },
    ]
  }
  if (props.eventType === 'competition') {
    return [
      { key: 'registered', label: 'Angemeldet' },
      { key: 'maybe', label: 'Vielleicht' },
      { key: 'canceled', label: 'Abgesagt' },
    ]
  }
  // training + social
  return [
    { key: 'yes', label: 'Dabei' },
    { key: 'maybe', label: 'Vielleicht' },
    { key: 'no', label: 'Nicht dabei' },
  ]
})
```

### Computed: Registrierungen gruppiert

```ts
const byStatus = computed(() =>
  Object.fromEntries(
    tabConfig.value.map(tab => [
      tab.key,
      props.registrations.filter(r => r.status === tab.key),
    ])
  )
)
```

### UTabs items-Array

```ts
const tabItems = computed(() =>
  tabConfig.value.map(tab => ({
    label: `${tab.label} (${byStatus.value[tab.key].length})`,
    key: tab.key,
  }))
)
```

### fullName-Helper

Bleibt unverändert:
```ts
function fullName(r: RegistrationDetail): string {
  return [r.firstName, r.lastName].filter(Boolean).join(' ') || 'Unbekannt'
}
```

## Abgrenzung: Was NICHT in diesem Redesign gemacht wird

- **Keine Admin-Funktionen** — LADV-Operationsstatus (`ladv_registered_at`, `ladv_canceled_at`) und Aktions-Buttons kommen in Session 9.9. Die Komponente wird jetzt so gebaut, dass man diese Infos später pro Listenzeile einfach ergänzen kann.
- **Keine Sortierung oder Filterung** — kein alphabetischer Sort, kein Suchfeld. Scope: saubere Gruppierung.
- **Keine Paginierung** — bei ~100 Mitgliedern und realistisch 3–15 Anmeldungen pro Event nicht nötig.

## Nuxt UI Referenz

- `UTabs` — Doku unter https://ui.nuxt.com/components/tabs
